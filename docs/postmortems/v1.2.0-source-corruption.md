# Postmortem: v1.2.0 Source Formatting Corruption

A postmortem analysis of the source formatting collapse incident in `intisari-starter` v1.2.0 and the recovery actions taken in v1.2.1.

## Summary

During release packaging or local synchronization of version v1.2.0, several source files in the repository appeared to lose their multiline structure or became collapsed into one-line fragments. This incident did not introduce syntax errors but affected general file readability and formatting validation.

## Impact

* **Readability:** Affected codebase files were collapsed into single-line formats.
* **Release Status:** Tag v1.2.0 was marked corrupted and required immediate superseding by hotfix v1.2.1.
* **CI/CD:** Required additional checks to be added to the GitHub Actions test workflow to detect future file formatting corruption.

## Symptoms

* Formatting checkers and linters encountered unexpected formatting structures.
* Code blocks in Markdown files and configurations showed collapsed one-line styles.
* Development team checks identified collapsed text lines inside `.env.example`, `phpunit.xml`, and various documentation files.

## Root Cause Hypothesis

* The source collapse was likely caused by a local workspace formatting/beautifier tool glitch, or a Git newline/formatting mismatch (e.g., CRLF/LF translation issues) occurring during local checkout or staging operations.

## Files Affected

* `composer.json`
* `.env.example`
* `phpunit.xml`
* `.github/workflows/tests.yml`
* `README.md`
* `docs/security/index.md`
* `docs/index.md`
* `docs/release-readiness.md`

## Fix Applied in v1.2.1

* **Restoration:** All codebase files were checked out fresh from pristine git history (HEAD / tag `v1.2.0` on the repository was verified healthy, confirming that the remote git history itself was not corrupted).
* **Escaping Fixes:** Refined HTML escaping examples and code block rendering inside `docs/security/index.md`.
* **Clean Routing:** Verified routes/web.php loadability and route handlers.
* **Link Audit:** Checked and validated that all links in the documentation are relative and contain no local path parameters.

## New Prevention Checks

* **Source Integrity Checker:** Implemented `scripts/check-source-integrity.php` and registered the `source:check` Composer command. This utility performs strict validation to prevent source collapse:
  * PHP file tags (`<?php`) and file non-emptiness.
  * `return` statements in configuration arrays.
  * Valid XML structure inside `phpunit.xml`.
  * Valid YAML configuration inside GitHub Action workflows.
  * Balanced multiline structures for `.env.example` and all Markdown documents.
* **CI Integration:** Configured the `source:check` validation check step inside `.github/workflows/tests.yml` to run automatically on every push and pull request.

## Release Checklist Updates

* Updated `docs/release-readiness.md` to list mandatory verification steps prior to tagging releases:
  * `composer source:check`
  * PHP syntax linting (`php -l`)
  * CI validation check

## Lessons Learned

1. **Automated Structure Validation:** Syntactically valid PHP/XML/YAML files can still suffer from physical structural collapse (e.g., one-line fragments). Standard test suites do not check for this, necessitating a dedicated structure integrity linter.
2. **Local Environment Safety:** Keep active formatters and IDE configurations aligned with project style guidelines to prevent accidental file degradation during sync or formatting operations.
